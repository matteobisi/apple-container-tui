# Implementation Plan: SBOM Generation for Binary Builds

**Branch**: `011-sbom-binary-build` | **Date**: 2026-04-04 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/011-sbom-binary-build/spec.md`

## Summary

Generate a Software Bill of Materials (SBOM) in SPDX 2.3 JSON format for every successful `actui` binary build, upload it as a workflow artifact, and attach it to every GitHub Release alongside the binary. Simultaneously pin the three bare-tag action references in `build-binary.yml` to commit SHAs. These two changes together materially improve the repository's OSSF Scorecard score via the `SBOM` check (0 → 10/10) and strengthen the `Pinned-Dependencies` score.

**Technical approach**: Add `anchore/sbom-action` (Syft) as a post-build step in `build-binary.yml`. Update `publish-release.yml` to download and attach the SBOM artifact. Update `docs/binary-build-automation.md`.

No TUI application source code changes. No new Go dependencies. No schema migrations.

## Technical Context

**Language/Version**: Go 1.24.2 (module `container-tui`)  
**Primary Dependencies**: `anchore/sbom-action` (Syft, composite GHA); existing: `actions/checkout`, `actions/setup-go`, `actions/upload-artifact`, `actions/download-artifact`  
**Storage**: GitHub Actions workflow artifacts (14-day retention) + GitHub Release assets (permanent)  
**Testing**: Manual verification: workflow run artifact inspection, release asset inspection, SPDX JSON format validation via `jq` or `pyspdxtools`  
**Target Platform**: GitHub Actions `ubuntu-latest` runner  
**Project Type**: CI/CD pipeline extension (GitHub Actions workflow modification)  
**Performance Goals**: SBOM generation adds ≤2 minutes to total build workflow duration (SC-006)  
**Constraints**: All actions MUST be pinned to immutable commit SHAs per AGENTS.md policy; `contents: read` permission must not be escalated for build workflow; no Node.js version constraints (anchore/sbom-action is a composite action)  
**Scale/Scope**: Two workflow files modified; one documentation file updated; no source code changes
