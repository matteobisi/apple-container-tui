# Data Model: SBOM Generation for Binary Builds

**Feature**: `011-sbom-binary-build`  
**Date**: 2026-04-04

## Overview

This feature introduces no new application entities, database schema, or persistent storage structures in the TUI application. All entities are CI/CD pipeline artifacts governed by GitHub Actions workflow definitions and GitHub's release storage system.

The data model documents the artifact schema, naming conventions, and relationships between the objects produced by the build pipeline.

---

## Entities

### 1. SBOM Document

**What it represents**: A machine-readable, standardised inventory of all software components included in the `actui` binary at the time of a specific build.

| Attribute | Value |
|---|---|
| Format standard | SPDX 2.3 JSON |
| File extension | `.spdx.json` |
| SPDX document namespace | Unique URI per build run (generated by Syft) |
| SPDX version | SPDX-2.3 |
| Data licence | CC0-1.0 (SPDX default) |
| Document name | `actui-linux-amd64` |
| Creator tool | `syft` via `anchore/sbom-action` |
| Created timestamp | ISO-8601 UTC timestamp of the workflow run |

**Component entries** (one per dependency):

| Field | Source | Notes |
|---|---|---|
| `SPDXID` | Generated | Unique identifier per component (e.g., `SPDXRef-Package-<hash>`) |
| `name` | `go.mod` / `go.sum` | Module path (e.g., `github.com/charmbracelet/bubbletea`) |
| `versionInfo` | `go.sum` | Exact version used (e.g., `v1.3.10`) |
| `downloadLocation` | Derived | Module proxy URL or `NOASSERTION` |
| `filesAnalyzed` | `false` | Standard for dependency-level SBOMs (not individual files) |
| `licenseConcluded` | From module metadata or `NOASSERTION` | Best-effort; not all Go modules embed licence data |
| `licenseDeclared` | From module metadata or `NOASSERTION` | |
| `supplier` | `NOASSERTION` | Module author attribution not always available |

**Relationships**:
- One SBOM Document is produced per binary build run
- One SBOM Document references exactly one Binary Artifact (the `actui` binary)
- One SBOM Document is uploaded as exactly one Workflow Artifact
- One SBOM Document is attached to at most one GitHub Release (as a Release Asset)

---

### 2. Binary Artifact (existing, extended)

**What it represents**: The compiled `actui` binary produced by `build-binary.yml`.

No schema changes to the binary artifact. The extension in this feature is that it gains a paired SBOM Artifact in the same workflow run.

| Attribute | Existing value |
|---|---|
| Artifact name (workflow) | `actui-linux-amd64` |
| Contents | Single executable file: `actui` → renamed `actui-linux-amd64` at release |
| Retention | 14 days (existing policy) |
| Release asset name | `actui-linux-amd64` |

---

### 3. SBOM Artifact (new)

**What it represents**: The workflow artifact produced in `build-binary.yml` that packages the SBOM Document for downstream use (release attachment).

| Attribute | Value |
|---|---|
| Artifact name (workflow) | `actui-linux-amd64-sbom` |
| Contents | Single file: `actui-linux-amd64.spdx.json` |
| Retention | 14 days (matches binary artifact retention) |
| Upload action | `actions/upload-artifact@ea165f8d65b6e75b540449e92b4886f43607fa02` (v4.6.2) |
| Download pattern (publish-release.yml) | Downloaded by name `actui-linux-amd64-sbom` |

---

### 4. Release Asset (extended)

**What it represents**: A file attached to a published GitHub Release.

Existing release has one asset. After this feature, each release has two assets:

| Asset name | Type | Source artifact |
|---|---|---|
| `actui-linux-amd64` | Binary executable | Workflow artifact `actui-linux-amd64` |
| `actui-linux-amd64.spdx.json` | SBOM Document (SPDX 2.3 JSON) | Workflow artifact `actui-linux-amd64-sbom` |

---

## Artifact Lifecycle

```text
build-binary.yml (run)
│
├─► SBOM Document (generated by Syft from go.mod/go.sum)
│     └─► uploaded as Workflow Artifact: actui-linux-amd64-sbom
│           └─► retained 14 days
│
└─► (downstream: publish-release.yml)
      ├─► downloads actui-linux-amd64 artifact
      ├─► downloads actui-linux-amd64-sbom artifact
      └─► publishes GitHub Release with both as Release Assets
            ├─► actui-linux-amd64           (binary, permanent)
            └─► actui-linux-amd64.spdx.json (SBOM, permanent)
```

## Naming Conventions

| Object | Name | Rationale |
|---|---|---|
| Workflow artifact (binary) | `actui-linux-amd64` | Existing; unchanged |
| Workflow artifact (SBOM) | `actui-linux-amd64-sbom` | Parallels binary; `-sbom` suffix distinguishes type |
| Release asset (binary) | `actui-linux-amd64` | Existing; unchanged |
| Release asset (SBOM) | `actui-linux-amd64.spdx.json` | `<binary-name>.spdx.json` per OSSF SBOM Everywhere naming guidance; `.spdx.json` extension is recognised by Scorecard SBOM check |
| SBOM file inside artifact | `actui-linux-amd64.spdx.json` | Consistent with release asset name |
