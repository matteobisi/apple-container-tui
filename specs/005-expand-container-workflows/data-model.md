# Data Model: Expanded Container Workflows

**Feature**: 005-expand-container-workflows  
**Date**: 2026-03-31

## Overview

This feature extends the current TUI with four domain-facing additions:
- runtime-managed registry login entries displayed in a new screen
- a stopped-container export workflow that materializes a local OCI tar archive through a command sequence
- a build option that controls `--pull`
- a richer daemon status result derived from structured JSON output

The design stays additive and keeps data responsibilities close to the existing `models` and `services` seams.

## Entities

### RegistryLogin

**Purpose**: Represent one registry login returned by `container registry list --format json`

**Fields**:
- `Hostname` (string, required): Registry host shown as the primary identifier
- `Username` (string, optional): Logged-in user or account shown as secondary context
- `CreatedDate` (int64, optional): Raw CLI timestamp value when present
- `ModifiedDate` (int64, optional): Raw CLI timestamp value when present

**Validation rules**:
- `Hostname` MUST be non-empty after trimming whitespace
- `Username` MAY be empty; UI falls back to a muted "no username" label if needed
- Raw timestamp fields MAY be zero or omitted without making the entry invalid

**Relationships**:
- Displayed by `RegistriesScreen`
- Produced by `RegistryParser`

**Display rules**:
- Primary row text is `Hostname`
- Secondary details line is `Username` when present
- Empty results map to screen-level empty state, not a synthetic registry entity

---

### ContainerExportPlan

**Purpose**: Represent one user-approved export request for a stopped container

**Fields**:
- `ContainerID` (string, required): CLI container identifier
- `ContainerName` (string, required): Human-readable name used for preview text and generated file naming
- `DestinationDirectory` (string, required): User-provided writable directory
- `GeneratedImageRef` (string, required): Temporary image reference used by `container export --image`
- `ArchivePath` (string, required): Final OCI tar path generated by the application
- `Commands` ([]Command, required): Ordered command sequence for export, save, and optional cleanup
- `State` (enum, required): `planned`, `running`, `succeeded`, `failed`, `succeeded_with_warning`
- `CleanupWarning` (string, optional): Warning message when archive creation succeeds but temporary-image cleanup fails

**Validation rules**:
- Export is valid only when the source container state is stopped
- `DestinationDirectory` MUST exist or be creatable according to selected UX rules; at minimum it must resolve to a writable directory before execution begins
- `GeneratedImageRef` and `ArchivePath` MUST be deterministic for the current request and MUST NOT be blank
- `Commands` MUST contain at least export and save steps; cleanup is optional but expected in normal flow

**State transitions**:
- `planned` -> `running` when the user confirms the preview
- `running` -> `succeeded` when export/save succeed and cleanup succeeds or is intentionally skipped
- `running` -> `succeeded_with_warning` when export/save succeed but cleanup fails
- `running` -> `failed` when export or save fails

**Relationships**:
- Created by `ContainerExportScreen`
- Orchestrated by `ExportWorkflowService`
- Uses `ImageDeleteBuilder` for cleanup when enabled

---

### BuildOptions

**Purpose**: Represent the build inputs needed by the existing build workflow after adding the `--pull` choice

**Fields**:
- `Tag` (string, required): Image tag supplied by the user
- `FilePath` (string, required): Selected Containerfile or Dockerfile path
- `ContextPath` (string, required): Build context directory
- `PullLatest` (bool, required): Whether `--pull` should be included; defaults to `true`

**Validation rules**:
- `Tag`, `FilePath`, and `ContextPath` reuse existing validation in `BuildImageBuilder`
- `PullLatest` always has a value; no tri-state behavior

**Relationships**:
- Edited in `BuildScreen`
- Consumed by `BuildImageBuilder`

**State transitions**:
- Initial screen state sets `PullLatest = true`
- User toggles `PullLatest`
- Submitted value becomes part of the previewed command and execution request

---

### DaemonStatusSnapshot

**Purpose**: Represent the daemon status shown in the TUI from structured JSON output

**Fields**:
- `State` (enum, required): `running`, `stopped`, or `unknown`
- `Running` (bool, derived/compatibility): `true` only when `State == running`
- `Version` (string, optional): API server version text when present
- `InstallRoot` (string, optional): Structured status metadata from the CLI
- `AppRoot` (string, optional): Structured status metadata from the CLI
- `LastChecked` (time, required): Timestamp of the latest parse
- `MissingFields` ([]string, optional): Required fields absent from the payload during classification

**Validation rules**:
- `State` MUST always be one of `running`, `stopped`, or `unknown`
- `Running` MUST be derived from `State`, not set independently
- Missing or malformed JSON yields `State = unknown`

**Relationships**:
- Produced by `ParseDaemonStatus`
- Displayed by `DaemonControlScreen`

**State transitions**:
- `unknown` on startup before first successful fetch
- `unknown` -> `running` or `stopped` when structured payload contains a recognized `status`
- Any known state -> `unknown` when required fields disappear or parsing fails

## Derived Naming Rules

### Generated Export Image Reference

**Pattern**: `actui-export/<container-slug>:<timestamp>`

**Rules**:
- Slug derived from container name when available, otherwise container ID
- Timestamp ensures uniqueness for repeated exports
- Reference MUST be valid for `container export --image`

### Generated Archive Path

**Pattern**: `<destination-directory>/<container-slug>-<timestamp>.oci.tar`

**Rules**:
- Uses the selected destination directory unchanged
- Filename generated by the app, not typed by the user
- `.oci.tar` suffix makes the export artifact format explicit

## View-State Notes

- Registries use the existing list-screen pattern: loading, loaded, empty, error.
- Export uses the existing input + preview + result pattern from `ImagePullScreen` and `BuildScreen`, but applies it to a command sequence.
- Daemon control retains the current action keys and adds explicit rendering for `unknown` state.

## Test-Relevant Invariants

- Running containers MUST NOT expose an export option in the submenu.
- Disabling `PullLatest` MUST remove `--pull` from the previewed build command.
- Missing `status` in daemon JSON MUST produce `unknown`, never `running` or `stopped`.
- Registry parsing MUST tolerate empty arrays and missing optional fields without crashing.
